# 10-layer multi-timeframe S/R zone system in Pine Script v6

A 494-line Pine Script v6 indicator implementing 10 independent support/resistance zone layers, each fed by its own `request.security()` call fetching pivot highs/lows, volume, and ATR from a configurable timeframe. Zones are created at swing points, validated by volume, sized by ATR, prevented from overlapping by a minimum-distance check, removed after configurable strike-based mitigation, and faded over time via alpha decay. The architecture places all `request.security()` calls at top-level scope and delegates zone logic to a reusable `processLevel` function.

**Watch for:** Centerlines (`drawCenterlines`) create new line objects every bar without deleting previous ones, rapidly exhausting the 500-line drawing limit and causing TradingView to auto-delete zone boundary lines via FIFO (confirmed). Volume validation checks the volume of the *confirmation* bar (swingLen bars after the actual pivot), not the pivot bar itself — a fundamental mismatch between what the filter intends to verify and what it actually measures (confirmed). The pivot value from `request.security()` can persist as non-na across multiple chart bars within the same HTF bar, potentially triggering duplicate zone creation attempts on consecutive bars despite the overlap check (likely).

## High-level view

The `request.security()` tuple pattern keeps total security calls to 10. However, `ta.pivothigh(high, N, N)` returns its value N bars *after* the actual swing, and the volume fetched in the same tuple is the volume of whichever HTF bar produced the pivot confirmation, not the bar at the swing point itself. The volume filter validates a bar that's temporally disconnected from the structural event it's supposed to confirm.

The drawing system redraws zone boxes and boundary lines every bar inside `processLevel` (delete-then-create). But `drawCenterlines` — called separately for each of the 10 layers — creates new lines unconditionally without deleting old ones. On every bar, up to 200 new line objects accumulate. TradingView's 500-line limit means older lines get silently garbage-collected, and since zone boundary lines (`topLine`, `botLine`) share the same pool, active zone boundaries will be deleted by the runtime once centerlines consume the budget.

For HTF layers (weekly/daily) viewed on a lower chart timeframe, `request.security()` with `lookahead_off` delivers the pivot value on the first chart bar after the HTF bar closes. Whether that value persists as non-na on subsequent chart bars depends on the internal evaluation semantics — if it does persist, the overlap check is the only defense against duplicate zones at the same price level.

<summary>Issues (4)</summary>

1. **Centerline line leak** — `drawCenterlines` creates new `line.new()` objects every bar without deleting previous ones. With up to 200 lines created per bar across 10 layers, the 500-line limit is exhausted within 2-3 bars, causing TradingView to FIFO-delete zone boundary lines (`topLine`/`botLine`) from `processLevel`. Store centerline references and delete them before redrawing, or add a `var` array to track them.

2. **Volume filter validates wrong bar** — `ta.pivothigh(high, N, N)` confirms a pivot N bars after the actual swing high. The `volume` value in the same `request.security()` tuple is the volume of the confirmation bar, not the pivot bar. To validate the pivot bar's volume, the expression inside security should be `volume[swingLen]` and `ta.sma(volume, 20)[swingLen]`.

3. **Touch alert fires intrabar, mitigation only on close** — Zone touch detection runs unconditionally while mitigation is gated behind `barstate.isconfirmed`. Both alerts can fire on the same bar — touch on the intrabar wick, mitigation on the close — which may confuse downstream automation expecting ordered state transitions. Consider gating touch detection behind `barstate.isconfirmed` as well, or documenting the expected alert sequence.

4. **Dead `var` alert state variables** — `alertZoneTouch`, `alertZoneMitigated`, and `alertNewZone` are declared as `var bool` but never read or written after declaration. Remove them.

## Centerline drawing object leak

```pine
drawCenterlines(bool enable, bool showCenterlines, ZoneData[] zones, color edgeColor) =>
    if enable and showCenterlines and array.size(zones) > 0
        for i = 0 to array.size(zones) - 1
            ZoneData zone = array.get(zones, i)
            float mid = (zone.priceHigh + zone.priceLow) / 2.0
            line.new(bar_index - 1, mid, bar_index, mid, xloc = xloc.bar_index, extend = extend.both, color = color.new(edgeColor, 50), style = line.style_dotted, width = 1)
```

Every bar, this creates one new line per zone per enabled layer. These lines are never stored in any reference variable and never deleted. TradingView maintains a global pool of 500 lines. Once the pool fills, the runtime deletes the *oldest* line objects — those are the `topLine` and `botLine` references stored in `ZoneData` from `processLevel`.

The result: zone boundary lines disappear after a few bars because the centerline function consumes the entire drawing budget. `processLevel` then re-creates boundary lines each bar, but those are immediately consumed by the next batch of centerlines. Boundary lines become effectively invisible once the indicator has been running for more than a handful of bars.

The fix is to store centerline references (either in the `ZoneData` UDT or a parallel `var` array) and delete them before creating new ones, mirroring the delete-then-create pattern already used for `boxRef`/`topLine`/`botLine` in `processLevel`.

## Volume validation temporal mismatch

```pine
[phL1, plL1, volL1, volSmaL1, atrL1] = request.security(syminfo.tickerid, tfL1, 
    [ta.pivothigh(high, swingLenL1, swingLenL1), ta.pivotlow(low, swingLenL1, swingLenL1), 
     volume, ta.sma(volume, 20), ta.atr(50)], lookahead = barmerge.lookahead_off)
```

`ta.pivothigh(high, N, N)` returns a non-na value on the bar that is N bars *after* the actual swing high (it needs N bars to the right to confirm the pivot). The `volume` in this tuple is the volume of *that confirmation bar*, not the volume at the swing point. The intent of volume validation is to confirm that the pivot formed with significant institutional participation — meaning the volume at the pivot bar itself should be above average. The current implementation checks whether volume was above average N bars later, which is structurally meaningless.

To fix, the volume expressions inside security should offset back to the pivot bar: `volume[swingLenL1]` and `ta.sma(volume, 20)[swingLenL1]`.

## Mitigation and touch alert timing asymmetry

Zone touch detection:
```pine
if high >= zone.priceLow and low <= zone.priceHigh
    touchFlag := true
```

This fires on every bar where the candle overlaps the zone, including unconfirmed (still-forming) bars. Mitigation:
```pine
if enable and useMitig and barstate.isconfirmed and array.size(zones) > 0
```

Only fires on confirmed bars. The downstream `alert()` with `alert.freq_once_per_bar` fires the touch alert on the first intrabar tick that overlaps the zone, potentially well before the bar closes. If the bar then closes through the zone, the mitigation alert fires at bar close. For a user with automation, the touch alert could fire many minutes or hours before the mitigation alert on the same bar depending on the chart timeframe. The semantics are technically correct (you touch before you breach) but the timing gap could cause confusion if automation assumes these are distinct events on separate bars.
