# Phantom Edge: 3-layer trading indicator in Pine Script v6

A 754-line Pine Script v6 indicator implementing a multi-timeframe trading system. Three layers cascade: Layer 1 derives directional bias from weekly/daily/4H pivot structure via 2-of-3 consensus; Layer 2 gates entry by checking lower-timeframe alignment (2H/1H/30M/15M); Layer 3 executes entries through a sweep-then-displacement confirmation sequence with S/R zones, liquidity pool detection, and FVG identification. The implementation uses `request.security()` with `barmerge.lookahead_off` throughout and gates entry signals behind `barstate.isconfirmed`.

**Watch for:** Previous-pivot tracking uses a flawed self-referencing update pattern that likely produces stale or self-assigned values, undermining the entire trend detection chain (confirmed). The `alertcondition()` calls use dynamic string concatenation in the `message` parameter which is invalid in Pine Script v6 — alert messages must be compile-time constants (confirmed). Sweep detection is coupled to the `showPDHL` display toggle, meaning disabling the visual also silences the trading logic (confirmed). Trade state machine never resets the `direction`, `entry`, `sl`, or `tp` fields on exit, leaving stale values visible in the dashboard between trades (confirmed).

## High-level view

The HTF bias system fetches pivot highs/lows from three timeframes and compares current pivots to "previous" pivots to determine trend. The previous-pivot tracking mechanism is the script's central structural defect: it attempts `nz(wPH[1], wPH)` inside a conditional that fires when the value has already changed, which means `prevWPH` often gets assigned the same value as the current pivot or a stale series reference, making the HH/HL / LH/LL comparison unreliable.

LTF alignment replicates the same pivot-tracking pattern for four additional timeframes. The trend detection feeds into a majority vote that gates entry signals. Because it inherits the same flawed previous-pivot logic, the "Ready" state may fire spuriously or fail to fire when the structure genuinely aligns.

The entry execution layer implements a well-structured 3-step sequence (sweep detection, displacement confirmation within a CISD window, then signal generation with calculated risk levels). The logic is sound in isolation but depends on the upstream layers being correct. Two operational issues stand out: the sweep check is gated behind `showPDHL` (a visual toggle), and the trade state machine leaks stale field values because it only resets `status` on exit without clearing price fields.

The alert system uses `alertcondition()` with dynamically-constructed strings, which will fail at compile time on TradingView. Pine Script requires alert messages to be string literals or simple constants.

<details>
<summary>Issues (7)</summary>

1. **Previous-pivot self-assignment** — The `if not na(wPH) and wPH != prevWPH` block assigns `prevWPH := nz(wPH[1], wPH)`. Since `wPH` just changed (that's the trigger condition), `wPH[1]` is the previous bar's security value which may already equal the "previous" pivot, or `nz` falls back to the current value. This likely produces `prevWPH == wPH` in many cases, making `detectTrend` return RANGING. Rework to store the *outgoing* value before overwriting: `prevWPH := wPH` should capture the old pivot, and the comparison should use that old value against the new one.

2. **Alert message dynamic strings** — `alertcondition()` in Pine Script v6 requires compile-time constant strings for the `message` parameter. The current code concatenates runtime variables (`htfBias == BULLISH ? "BULLISH" : ...`, `entryAction`, etc.) which will cause a compilation error. Use TradingView placeholders (`{{ticker}}`, `{{close}}`) or static messages, and move dynamic data to `alert()` function calls inside `if` blocks instead of `alertcondition()`.

3. **Sweep detection gated by display toggle** — `sweepHigh` and `sweepLow` conditions include `showPDHL` as part of the sweep logic (`if (showPDHL and high > pdh ...)`). If a user disables PDH/PDL visuals, sweep detection against those levels stops entirely. Decouple the visual toggle from the logic.

4. **Trade state partial reset** — When TP1 or SL is hit, only `tradeState.status` is set to "Idle". The `entry`, `sl`, `tp1`, `tp2`, `tp3`, and `direction` fields retain their last values. The dashboard conditionally displays these based on status, but any code path that reads `tradeState.direction` or `tradeState.entry` outside the dashboard (e.g., alert construction uses `tradeState.direction`) could reference stale data.

5. **`request.security` with `high[1]`/`low[1]` for PDH/PDL** — The call `request.security(syminfo.tickerid, "D", [high[1], low[1]], lookahead = barmerge.lookahead_off)` fetches the *previous bar's* high/low within the daily context. With `lookahead_off`, this returns the prior-prior day's values during most of the current day (delayed by one extra period). The intended PDH/PDL should reference the *completed* prior day's high/low. The standard anti-repaint pattern is `request.security(syminfo.tickerid, "D", [high[1], low[1]], lookahead = barmerge.lookahead_on)` or fetching `high`/`low` with `lookahead_off` and indexing `[1]` on the result.

6. **EQH/EQL detection only checks newest pivot against all older** — The loop compares only `array.get(recentHighs, size-1)` against earlier entries. If two older pivots form equal highs but the latest doesn't match, the EQH is missed. This is a design limitation that reduces detection coverage.

7. **S/R zone array grows without time-based expiry** — Zones accumulate until they hit `maxZones`, at which point the oldest is shifted off regardless of its relevance. There is no proximity-to-price or staleness check, so zones far from current price consume slots while nearby relevant structure may be evicted.

## Previous-pivot tracking and trend detection

The entire indicator's directional logic flows from `detectTrend()`, which compares a current pivot high/low pair against a "previous" pair. The tracking mechanism at lines ~168-179:

```pine
if not na(wPH) and wPH != prevWPH
    prevWPH := nz(wPH[1], wPH)
```

The intent is: "when a new weekly pivot high appears, store the old one." But `wPH[1]` in Pine Script's execution model refers to the previous bar's value of the `wPH` series — which, for `request.security` data, is the same pivot value that was already assigned to `prevWPH` last time it changed, or `na` if no pivot was detected on the prior bar. The `nz(..., wPH)` fallback then assigns the *current* new pivot as its own "previous," making `detectTrend` compare equal values and return RANGING.

The correct pattern is:

```pine
if not na(wPH) and wPH != prevWPH
    prevWPH := prevWPH  // no — this is a no-op; need two-level tracking
```

What's actually needed is a two-level shift: store the outgoing `prevWPH` into a `prevPrevWPH` if needed, or simply: `prevWPH := wPH` and compare `wPH` (new) against `prevWPH` (which still holds the old value *before* this assignment). The current code overwrites `prevWPH` with a value derived from `wPH[1]` which is unreliable in the security context.

This same pattern is replicated for all 7 timeframes (weekly, daily, 4H, plus 2H, 1H, 30M, 15M), making the structural defect systemic.

## PDH/PDL/PWH/PWL level accuracy

```pine
[prevDayHigh, prevDayLow] = request.security(syminfo.tickerid, "D", [high[1], low[1]], lookahead = barmerge.lookahead_off)
```

With `lookahead_off`, `request.security` returns the value from the *last confirmed* higher-timeframe bar. On the daily timeframe, that's the current day's data once it closes, or the previous day's data while intraday. But the expression inside is `high[1]` — one bar back *within* the daily context. So during the current trading day, this returns the high of the day *before yesterday*. The levels labeled PDH/PDL are actually the day-before-yesterday's range, not yesterday's.

The standard approach for yesterday's high/low without repainting is either:
- `request.security(sym, "D", high[1], lookahead = barmerge.lookahead_on)` — uses lookahead to get the *completed* prior bar
- Fetch `high` with `lookahead_off` and index `[1]` on the *result*: `float dHigh = request.security(..., high, lookahead_off)` then use `dHigh[1]` — but this introduces a one-bar delay on the chart timeframe, not the daily timeframe

The same issue applies to the PWH/PWL security call.

## Alert system compilation failure

Pine Script v6's `alertcondition()` requires the `message` parameter to be a "simple string" — a value determinable at compile time. The current implementation:

```pine
alertcondition(biasChanged and alertBiasChange, title = "HTF Bias Changed", 
    message = "Phantom Edge: HTF Bias changed to " + (htfBias == BULLISH ? "BULLISH" : ...) + " on {{ticker}}")
```

The ternary expression `htfBias == BULLISH ? "BULLISH" : ...` is a *series* string (changes bar-to-bar), which violates the compile-time requirement. This will produce a compilation error when the script is loaded in TradingView. The `{{ticker}}` placeholder is valid, but the dynamic portions are not.

The entry signal alert has the same issue with `entryAction`. The `jsonPayload` variable is constructed but never used in any alert — it's dead code.

## Sweep detection's dependency on visual toggles

The sweep condition at line ~474:

```pine
if (showPDHL and high > pdh and close < pdh) or (eqhDetected and ...)
    sweepHigh := true
```

`showPDHL` is the "Show PDH/PDL" checkbox in the Visuals input group. A user who hides the lines to reduce chart clutter also disables half the sweep detection logic. The EQH/EQL path still works, but PDH/PDL sweeps — typically the most frequent liquidity targets — are silenced. The visual toggle should control drawing only, not logic evaluation.

## Trade state lifecycle gaps

When `tradeState.status` transitions to "Idle" on TP1/SL hit, only the `status` field is reset. The `entry`, `sl`, `tp1-3`, `direction`, and `signalBar` fields retain their values from the last trade. This creates two issues:

The dashboard correctly gates display on `status == "Entry Active"`, so stale values don't show there. But the alert construction reads `tradeState.direction` unconditionally when building `entryAction`:

```pine
string entryAction = entrySignal ? (tradeState.direction == BULLISH ? "BUY" : "SELL") : ""
```

If `entrySignal` fires on a new setup, `tradeState.direction` has already been updated by the sweep detection block, so this particular path is safe. However, the Entry/SL/TP lines drawn on the chart (`lineEntry`, `lineSL`, `lineTP1`) are never deleted when the trade exits — they persist with `extend.both` until the next entry overwrites them. This leaves misleading trade levels on the chart indefinitely after a trade closes.

## CISD window race condition

The displacement check runs on every bar within the CISD window:

```pine
if not na(sweepBar) and bar_index - sweepBar <= cisdWindow and bar_index > sweepBar
```

But the FVG confirmation path (`if useFVG ... if sweepDirection == BEARISH and bearishFVG`) doesn't verify that the FVG *direction* aligns with the displacement requirement. A bullish FVG appearing during a bearish sweep's CISD window sets `cisdConfirmed = true`, which would generate a short entry confirmed by a bullish gap — logically contradictory. The FVG check should be direction-matched (it partially is — `bearishFVG` for `BEARISH` sweep is correct), but the issue is that FVG alone, without a displacement candle, triggers confirmation. The spec says CISD *or* FVG is sufficient, which this implements, but a stray gap unrelated to the sweep context could false-trigger.

## Zone management performance on every bar

The S/R zone drawing block deletes *all* zone boxes and redraws them every bar:

```pine
for i = 0 to array.size(zoneBoxes) - 1
    box.delete(array.get(zoneBoxes, i))
array.clear(zoneBoxes)
```

With up to `maxZones` (default 10, max 50) zones, this creates and destroys up to 50 boxes per bar. Pine Script handles this but it's inefficient and can cause flickering on fast-updating charts. A check for whether zones actually changed would reduce unnecessary redraws.
