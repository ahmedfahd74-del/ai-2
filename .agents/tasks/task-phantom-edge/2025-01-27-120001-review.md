# Phantom Edge: v2 fix verification and residual concerns (v2)

All seven issues from the v1 review have been addressed with correct implementations. The previous-pivot tracking now uses a proper two-variable shift pattern, alertcondition messages are compile-time constants, PDH/PDL uses the standard `lookahead_on` + `high[1]` anti-repaint pattern, sweep detection is decoupled from the visual toggle, trade state fully resets on exit including line deletion, EQH/EQL uses nested pair comparison, and zone eviction uses distance-from-price. The script grew from 754 to 852 lines, mostly from the expanded reset logic and nested EQH/EQL loops.

**Watch for:** The `nz()` wrapper on `prevWPH`/`prevWPL` (and all equivalent prev variables) defeats the na-guard inside `detectTrend`, causing false trend signals before a second pivot forms (confirmed). Three declared inputs (`htfLookback`, `useRetest`, `useDynamicTargets`) and one UDT (`PivotData`) are dead code — they appear in the input panel but affect nothing (confirmed). PWH/PWL levels are drawn but never checked in sweep detection, reducing the liquidity sweep surface (confirmed).

**Verdict**: NEEDS_CHANGES

## High-level view

The two-variable pivot shift (`prevWPH := currentWPH; currentWPH := wPH`) is the correct structural fix for the central v1 defect, and the same pattern is consistently applied across all 11 timeframes. However, the `detectTrend` call sites wrap every argument in `nz()`, converting an uninitialized `na` into `0.0`. Since `detectTrend` guards against `na` but not against zero, the first pivot on each timeframe will produce a spurious BULLISH or BEARISH reading because any real price is greater than zero. This is a warm-up transient that resolves once two pivots have formed, but on weekly timeframes that transient can last weeks of chart bars.

The entry execution chain is correctly gated — sweep fires only when bias and alignment agree, CISD/FVG confirmation is directionally matched, and the trade state machine fully resets all fields plus chart lines on exit. But sweep detection checks PDH/PDL and EQH/EQL levels while ignoring PWH/PWL, despite those levels being fetched and drawn as presumably meaningful weekly liquidity targets.

1. **nz() masks uninitialized prev-pivots** — `detectTrend(nz(currentWPH), nz(currentWPL), nz(prevWPH), nz(prevWPL))` converts the initial `na` state of prev variables to `0.0`, bypassing the na-guard inside `detectTrend`. 2. **PWH/PWL excluded from sweep detection** — Sweep logic checks `high > pdh` and `low < pdl` for daily levels, and EQH/EQL for liquidity pools, but never checks `high > pwh` or `low < pwl`. Weekly highs/lows are prominent liquidity targets. Add PWH/PWL sweep conditions alongside the PDH/PDL checks.

3. **Dead inputs and dead type** — `htfLookback`, `useRetest`, and `useDynamicTargets` are declared as inputs but never referenced in logic. The `PivotData` UDT is declared but never instantiated. Remove or implement them to avoid confusing users with non-functional controls.

4. **signalBar not reset on trade exit** — The trade reset block clears `entry`, `sl`, `tp1-3`, `direction` but not `signalBar`. Minor, since nothing currently reads `signalBar` after exit, but it's an inconsistency that could bite if future code checks it.

## nz() vs na-guard interaction in detectTrend

The `detectTrend` function correctly returns RANGING when any of its four arguments is `na`:

```pine
detectTrend(float ph, float pl, float prevPH, float prevPL) =>
    int result = RANGING
    if not na(ph) and not na(pl) and not na(prevPH) and not na(prevPL)
        ...
```

But every call site wraps arguments in `nz()`:

```pine
int weeklyTrend = useWeekly ? detectTrend(nz(currentWPH), nz(currentWPL), nz(prevWPH), nz(prevWPL)) : RANGING
```

`nz(prevWPH)` returns `0.0` when `prevWPH` is `na` (its `var` initial state). So `detectTrend` receives `(somePrice, somePrice, 0.0, 0.0)` and evaluates `somePrice > 0.0` → true for both higherHigh and higherLow → returns BULLISH. This persists on every bar until a second distinct pivot forms on that timeframe. For the weekly, that means the indicator shows a spurious BULLISH weekly trend for potentially weeks of real-time operation after being added to a chart.

The fix is to remove `nz()` from these call sites. The na-guard already handles the case correctly by returning RANGING.

## PWH/PWL as sweep targets

The sweep detection block (lines 529-537) checks three liquidity sources: PDH, PDL, EQH, and EQL. PWH/PWL are fetched (line 445), stored as `pwh`/`pwl`, and drawn as horizontal lines, but are never tested in the sweep condition. A wick above the previous week's high that closes back below is a classic weekly liquidity sweep, arguably more significant than a daily one. The omission means the indicator cannot trigger entries off weekly structure sweeps despite having the data available.
