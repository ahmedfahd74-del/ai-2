# Phantom Edge: final review after v2 fixes (v3)

An 847-line Pine Script v6 indicator implementing a 3-layer directional trading system. All four issues from the v2 review have been correctly resolved: `nz()` wrappers removed from `detectTrend` call sites, PWH/PWL added to sweep detection, dead inputs and the unused `PivotData` type removed, and `signalBar` now resets with the other trade state fields. The script compiles cleanly as far as static analysis can determine — no remaining type errors, no dynamic strings in `alertcondition`, no logic gated behind visual toggles.

**Watch for:** No blocking issues remain. Two minor observations below (zone redraw churn, EQH/EQL scope) are design-level trade-offs, not bugs.

**Verdict**: APPROVED

## High-level view

The pivot tracking system now uses the correct two-variable shift pattern (`prevWPH := currentWPH; currentWPH := wPH`) across all 11 timeframes, and passes raw `na`-capable values into `detectTrend`. The function's existing `not na(...)` guard returns RANGING until two distinct pivots populate — no false signals during the initial bars.

Sweep detection checks PDH, PDL, PWH, PWL, EQH, and EQL. The sweep condition is fully independent of display toggles (`showPDHL`/`showPWHL` control only line drawing). The trade state machine fully resets all eight fields and deletes chart lines on TP1/SL exit.

Alerts split between `alertcondition()` (static messages with `{{ticker}}`/`{{close}}` placeholders) and `alert()` inside `if` blocks for dynamic JSON payloads.

## v2 fix verification

The `nz()` removal is the most impactful fix. The call sites now read:

```pine
int weeklyTrend = useWeekly ? detectTrend(currentWPH, currentWPL, prevWPH, prevWPL) : RANGING
```

When `prevWPH` is still `na` (no second pivot yet), `detectTrend` hits `not na(prevPH)` → false → returns RANGING. Previously, `nz(prevWPH)` converted `na` to `0.0`, making any real price appear as a "higher high" and producing a spurious BULLISH reading for weeks.

PWH/PWL sweep conditions now include weekly levels alongside daily and EQH/EQL:

```pine
if (high > pdh and close < pdh) or (eqhDetected and not na(eqhLevel) and high > eqhLevel and close < eqhLevel) or (high > pwh and close < pwh)
```

## Design observations (non-blocking)

The S/R zone rendering deletes all boxes and redraws them every bar. With up to 50 zones this creates churn in the drawing engine. Not a correctness issue, but on fast-updating charts with large `maxZones` values, this could contribute to visual flicker. A dirty-flag check would eliminate unnecessary redraws.

EQH/EQL detection compares the last 5 entries in the recent-pivots array against each other. This won't detect equal levels between a pivot from 15 entries ago and one from 3 entries ago if neither falls in the last-5 window. The 0.1% tolerance and 20-entry rolling window are reasonable constraints for practical use.
